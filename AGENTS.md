# AGENTS.md — инструкции для ИИ-ассистентов

## Правила работы

- Если задачу пользователя можно не делать и получить результат — расскажи как
- Код меняем только в том случае, если без этого не получить результат
- Если пользователь предлагает сложную правку — попробуй упростить её, если это возможно
- Если пользователь что-то упускает или не прав — укажи на проблему и предложи решение
- KISS и DRY — главные принципы
- Основная цель — краткий и максимально читаемый код с хорошим форматированием
- Старайся не просто показать решение, а объяснить, почему именно это решение выбрано и в чём его преимущество перед альтернативами

## Особенности проекта

- **Русский язык во всём проекте.** Весь UI, комментарии, логи, сообщения — на русском.
  Исключение: общепринятые названия (JUCE, VST3, MIDI, JSON, plugin, host),
  названия файлов, технические идентификаторы (классы, функции, переменные).

- **Единая точка обращения к данным.** По возможности не хардкодить переменные и списки.
  Все цвета — в `Theme.h`, настройки — через `SettingsManager`, пути — через
  `Juce::File::getSpecialLocation()` или переменные окружения. Никаких `C:\пути\хардкодом`.

- **Версия — один источник истины.** Номер версии задаётся только в `CMakeLists.txt`
  (`project(SoLyP VERSION x.y.z ...)`), в коде используется через `SOLYP_VERSION`.
  Никаких хардкодов версии в исходниках.

- **Автосохранение настроек.** Все имеющиеся и будущие настройки должны автосохраняться
  при изменении (`SettingsManager::save()`) и автовосстанавливаться при открытии
  (`SettingsManager::load()`). Файл: `Documents/SoLyP/settings.json`.

- **Многоязычность.** Проект должен быть готов к локализации с самого начала.
  Языковые файлы — `Documents/SoLyP/langs/*.json` (UTF-8 без BOM).
  При проектировании новых строк учитывать склонения (одно яйцо, два яйца, много яиц).

- **Публикация релизов на GitHub.** Использовать `gh` CLI через `cmd.exe`,
  минуя PowerShell, чтобы избежать проблем с кодировкой кириллицы.
  Тело релиза — в файле (UTF-8 без BOM), флаг `--notes-file`.

### Кодировка

- **Все файлы проекта** (.cpp, .h, .json, .md) — только **UTF-8 без BOM**.
  Никаких ANSI, CP1251, UTF-16.

- **MSVC-флаг /utf-8.** В `CMakeLists.txt` обязательно указывать для MSVC:
  ```cmake
  if(MSVC)
      target_compile_options(SoLyP PRIVATE /W4 /utf-8)
  endif()
  ```
  Без этого флага MSVC читает кириллицу в .cpp как Windows-1251,
  и строки с русским текстом превращаются в кракозябры.

- **Локали — отдельные JSON-файлы.** Строки UI не хранить в C++ raw string
  literals, если там есть не-ASCII (русские буквы). Вместо этого:
  - `Documents/SoLyP/langs/ru.json` — русская локаль
  - `Documents/SoLyP/langs/en.json` — английская локаль
  - Файлы — UTF-8 без BOM.
  - Если файл локали не найден — использовать встроенный fallback
    (английский, без кириллицы в исходнике).

- **JUCE String из UTF-8.** При создании `juce::String` из `const char*`,
  содержащего не-ASCII (русские буквы), использовать `String::fromUTF8()`.
  Конструктор `String(const char*)` использует системную кодировку
  (Windows-1251 на русской Windows), что приводит к кракозябрам.

## Нейминг

### Переменные, классы, функции

- Имя должно отражать **назначение**, а не значение:
  - ✅ `buttonStart`, `textPrimary`, `sliderTrack`
  - ❌ `buttonBlue`, `textDark`, `sliderGrey`
- **Запрещены взаимные назначения** (одна переменная ссылается на другую). Каждая
  переменная хранит собственное значение:
  - ✅ `editorBg = Colour(0xFF13294B)`, `editorText = Colour(0xFFE0EDFF)`
  - ❌ `editorBg = bgPanel`, `editorText = textPrimary`
- Если одна переменная логически равна другой — обе должны быть явно объявлены
  с одинаковым значением. Исключение: только с явного разрешения разработчика.

### Файлы

- Формат: `PascalCase` (C++-стиль):
  - ✅ `PluginEditor.cpp`, `ControlsPanel.h`, `Song.cpp`
  - ❌ `plugin_editor.cpp`, `controls_panel.h`
- Имя отражает суть содержимого — группа функций, связанных единым контекстом
- Имя не должно быть длинным. Если контекст укладывается в одно слово — одно слово
- Для группы однотипных файлов: общий контекст + специализация:
  - ✅ `Song.h`, `SettingsManager.h` (контекст в названии)
  - ✅ `TabStatus.h`, `TabSettings.h`, `TabLogs.h`
  - ❌ `StatusTab.h`, `SettingsTab.h`, `LogsTab.h`
- **Запрещены файлы-двойники** с одинаковым именем (без учёта пути) во всём проекте:
  - ❌ `Source/UI/audio.h` + `Source/Data/audio.h`
  - Если имя занято — дать более конкретное, отражающее путь или контекст

### Пути

- Не хардкодить пути. Использовать:
  - `%USERPROFILE%`, `%LOCALAPPDATA%`, `%APPDATA%`
  - `Juce::File::getSpecialLocation(Juce::File::userDocumentsDirectory)`
  - ✅ `Documents/SoLyP/songs/`
  - ❌ `C:\Users\username\Documents\SoLyP\songs\`

### Списки

- Если элементов в списке больше 3 — предлагать разработчику вынести в отдельный JSON

## Размер файлов

- **Норма:** ≤350 строк на файл
- **Допустимо:** ≤525 строк, если:
  - контекст файла един и целостен,
  - код не подразумевает переиспользование из других мест,
  - и при этом **не** создаются вспомогательные файлы
- **Превышение 525 строк** — дробить:
  1. **Шаг 1** — выделить чистые хелперы в `{Имя}Helpers.h/.cpp`.
     После этого **все** файлы группы — ≤350 строк.
  2. **Шаг 2** — если после шага 1 всё ещё >350 — превратить в директорию
     `{контекст}/` с файлами `{Имя}Main.cpp`, `{Имя}Render.cpp`,
     `{Имя}Handlers.cpp`, `{Имя}Helpers.cpp`.
     Каждый файл в директории — ≤350 строк.

## GitHub

- **Пуш на GitHub только после явного подтверждения пользователя.** Никогда не пушить без разрешения.
- Если пользователь давно не пушил — напомнить и спросить, но не пушить самому.
- Токен: `%USERPROFILE%\.github_token` (ограничен по времени).
- Для публикации релизов с кириллицей — `gh` CLI через `cmd.exe`, тело в файле UTF-8 без BOM.

## Порядок описания структуры проекта

При описании структуры проекта придерживаться следующего порядка:

1. Сначала скрытые папки (с точкой в начале) в алфавитном порядке (регистр не важен)
2. Затем обычные папки в алфавитном порядке (регистр не важен)
3. Затем скрытые файлы (с точкой в начале) в алфавитном порядке (регистр не важен)
4. Затем обычные файлы в алфавитном порядке (регистр не важен)
5. Для вложенных папок/файлов — тот же принцип рекурсивно
6. Содержимое скрытых папок (с точкой в начале) не расписывать — указать только саму папку
